# SwingIQ — Agent Handoff Briefing

> Last updated: 2026-06-26. This file is the single source of truth for another AI agent picking up this project. Read it fully before changing code.

## What This Project Is
**SwingIQ** is a golf-swing AI analysis web app. A user uploads a slow-motion video of their swing (front/face-on and/or back/down-the-line), marks the swing window, picks their club, and gets a plain-language coaching report generated by Claude. Three specialist AI "coaches" analyze the swing in parallel, then a "head coach" synthesizes a final report. A separate Haiku model picks 9 key frames to display.

## Tech Stack
- **Backend:** Python / Flask, served by gunicorn (2 workers) inside Docker
- **AI:** Anthropic Claude API (`anthropic` Python SDK)
  - 3 specialist agents → `claude-opus-4-8`
  - Head coach synthesis → `claude-opus-4-8` with `thinking={"type":"adaptive"}`
  - Frame selector → `claude-haiku-4-5`
  - **Prompt caching** is on: the shared image prefix (~50 frames) is cached once and reused across all 4 Opus calls. First agent runs alone to warm the cache, then the other 2 + head coach run.
- **Computer vision:** OpenCV (`opencv-python-headless`) + MediaPipe Tasks API (pose skeleton + joint angles)
- **Frontend:** Server-rendered Jinja templates, vanilla JS, no framework
- **Persistence:** SQLite locally, **PostgreSQL on Railway** (via `DATABASE_URL`) for free-usage tracking. History + settings are per-device in browser `localStorage`.

## Routes (app.py)
- `GET /` → index.html
- `GET /history` → history.html
- `GET /settings` → settings.html
- `GET /usage-status` → JSON `{free_remaining, free_limit}`; sets the device cookie
- `GET /debug-status` → JSON sanity check (mediapipe loaded?, model file present?, server key length)
- `POST /analyze` → the main pipeline (see below)
- All HTML responses get `Cache-Control: no-cache` via `@app.after_request` so UI changes show immediately.

## The /analyze Pipeline (in order)
1. Read video file(s) + form fields: `club`, `swing_start_sec`, `swing_end_sec`, `api_key`, `email`.
2. **Access check:**
   - If `api_key` provided → validate it, use it (unlimited).
   - Else (free use) → **require a valid, non-disposable email**, then `consume_free_analysis(ip, device, email)`. Returns HTTP 402 if the daily free limit is used up.
3. `extract_frames()` per view:
   - Uses the user-marked window (`swing_start_sec`/`swing_end_sec`) if provided, else auto-detects swing start via motion differencing.
   - **Sampling:** frame 1 = address (just before start); **every 10th frame** through the backswing until the **top of backswing** (detected as peak `lead_arm_elevation` from MediaPipe); then **every frame** from the top to the end, capped at 40.
   - Draws MediaPipe skeleton + burns "FRAME N" label; returns `(frames_b64, pose_summary, all_angles)`.
4. Build a shared, cached image prefix (intro text + all frames, `cache_control: ephemeral` on the last block).
5. **3 specialist agents** (Body & Posture, Club & Hands, Tempo & Rhythm) run on Opus. First runs alone (warms cache), then the other two in parallel.
6. **Head coach** (Opus, adaptive thinking) synthesizes a structured markdown report. Retries up to 3× if Claude returns empty. Report sections: Quick Summary, Where the Coaches Agree, Where They Differ, The Complete Issue List (Major/Moderate/Minor), What's Working Well, Suggested Practice Order.
7. **Frame selector** (Haiku 4.5) picks the 9 key positions (Address → Follow-Through) using the pose-angle table; top-of-backswing pick is force-anchored to peak lead-arm elevation.
8. Returns JSON: `{analysis, frames:[{view, phase, frame, data}], agent_reports:[{name, report}]}`.
9. Any error along the way refunds the free analysis (`refund_free_analysis`).

## Free-Usage Tracking (important)
- **Why it's not in-memory:** gunicorn runs 2 workers, each with its own memory — an in-memory counter let users get extra free scans on reload. Now uses a shared DB.
- **Storage:** `_count()` / `_bump()` auto-select Postgres (if `DATABASE_URL` is set) or SQLite. Table `usage(key, day, count)`.
- **Keyed by three identities:** `email:<normalized>`, `dev:<cookie uuid>`, `ip:<client ip>`. A scan is blocked if ANY of them has hit `FREE_LIMIT` (1/day). All three are bumped on consume.
- **Email anti-abuse:** `validate_email()` blocks ~50 disposable domains (`DISPOSABLE_DOMAINS`). `normalize_email()` strips `+tags` for all domains and dots for Gmail so aliases collapse to one identity.
- **Railway setup:** A PostgreSQL service is attached and `DATABASE_URL` is referenced into the web service's Variables. Startup logs show `Usage tracking: PostgreSQL (persistent)`.

## Frontend Notes
- **Email gate:** Clicking "Analyze My Swing" opens a popup modal asking for email (free users only; API-key users skip it). Email saved to `localStorage['swingiq_email']`.
- **Swing window selector:** After upload, a scrubber with draggable START/END handles lets the user mark the swing; sends `swing_start_sec`/`swing_end_sec`.
- **History:** Each result saved to `localStorage['swingiq_history']` (last 10). `/history` lists and replays them. Per-device only (no accounts).
- **Settings:** `/settings` has Profile, Account, Appearance (light/dark/system theme), Accessibility (text size, reduce motion, high contrast), Notifications (saved prefs only — marked "COMING SOON", nothing actually sends yet). Prefs are stored as `swingiq_*` localStorage keys and applied via `data-theme`/`data-fontsize`/`data-motion`/`data-contrast` on `<html>` (see `_prefs_script.html` + dark/a11y CSS in `_styles.html`).
- All `swingiq_*` localStorage keys: `api_key, email, name, handicap, default_club, history, theme, fontsize, reducemotion, contrast, notif_results, notif_tips, notif_product, device(cookie)`.


## Cost (rough, per swing)
~$0.55–0.70 single angle with prompt caching (was ~$1.35 before caching). ~2× for two angles. Opus on the 4 calls dominates; Haiku selector is ~$0.05.

## Known Errors Already Solved (don't reintroduce)
- OpenCV needs `opencv-python-headless` + system libs (`libxcb1, libgl1, libgles2, libegl1, ...`) → handled in `Dockerfile`. Standard `opencv-python` crashes with `libxcb.so.1` missing.
- MediaPipe must use the **Tasks API** (`pose_landmarker.task` model), not the removed `solutions` API.
- `.env` loaded with `override=True` to beat stale system env vars.
- Opus 4.8 only supports `thinking={"type":"adaptive"}` (no `budget_tokens`, no `temperature`).
- Empty-response retries on the head coach call.

# ⭐ NEW TASK TO BUILD: YouTube Video Recommendations

**Goal:** After the head coach produces the issue list, the head coach (or a dedicated step) should find **helpful YouTube videos that correspond to the specific issues found in the swing**, and surface them in the output so the golfer can watch drills/instruction for their actual faults.

### Desired behavior
- For each MAJOR (and ideally MODERATE) issue in the report, provide 1–2 relevant YouTube instructional videos (title + link, ideally a 1-line "why this helps").
- Display them in the results UI — e.g. a new "📺 Recommended Videos" section under the analysis, and include them in saved history.

### Implementation options (pick one; recommendation first)

**Option A — Anthropic web search server tool (recommended).**
Give the synthesis (or a follow-up Claude call) the `web_search` server tool so Claude can search the web/YouTube and return real, current video links with citations. Cleanest because Claude picks contextually relevant videos and returns real URLs.
- Read the bundled `claude-api` skill (server-side tools / web search) for the exact tool block + how to read `web_search` results and citations. **Do not guess the API shape — confirm it from the skill or the SDK docs.**
- Likely a separate, cheap call AFTER the head coach report: pass the final issue list as text, enable web search, ask for YouTube links per issue. Return structured JSON.
- Watch latency/cost — this adds a tool-use round trip.

**Option B — YouTube Data API v3.**
Have Claude output a short search query per issue (it already names the issues), then call the YouTube Data API server-side to fetch the top video for each query. Returns verified, embeddable videos with thumbnails.
- Needs a `YOUTUBE_API_KEY` env var (Google Cloud). Add to Railway Variables.
- More moving parts but gives real metadata (title, channel, thumbnail, viewcount) for a nicer UI.

**Option C — Search-query links (simplest, no new keys).**
Have the head coach output a YouTube **search URL** per issue (`https://www.youtube.com/results?search_query=...`). Zero API cost, but links to a search page, not a specific verified video. Good MVP if you want to ship fast.

### Suggested data flow (Option A/B)
1. Head coach returns the report as today.
2. New function `find_videos(ai_client, issue_titles: list[str]) -> list[{issue, title, url, why}]`.
3. Add `videos` to the `/analyze` JSON response.
4. Frontend: render a "Recommended Videos" card in `displayResult()` (in BOTH `index.html` and `history.html` — they each have their own copy of `displayResult`). Include `videos` in the object saved to history so they persist.

### Gotchas
- `displayResult` is duplicated in `index.html` and `history.html` — update both, or refactor into a shared `static/render.js`.
- Keep the videos call resilient: if it fails, still return the analysis (don't fail the whole request). Refund logic should not be affected.
- Validate/sanitize any URLs before rendering as links.
